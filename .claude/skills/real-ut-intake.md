# 실 사용자 UT 세션 인테이크 (Real UT Intake)

> **목적**: 실제 사용자가 진행한 세션의 원본 데이터를, AI 시뮬레이션 결과(`UT_OBSERVATION_SHEET.md`)와
> **필드 단위로 비교 가능한 표준 스키마**로 정규화한다. 이 스킬은 세션을 만들지 않는다 —
> 이미 진행된 세션의 결과를 구조화하는 것만 담당한다.
>
> **역할 경계**: 참가자 모집·동의서·스크리닝·세션 진행(모더레이션)은 이 스킬의 범위 밖이다.
> 그건 리서치 패널·외부 툴(Maze/UserTesting/Dovetail 등)·사내 리서치팀이 담당하고,
> 그 결과물(녹화본·노트·타임코드)이 **이미 존재한다는 전제**에서 이 스킬이 시작된다.

---

## 진입 조건 (Step 0 게이트)

다음 중 최소 하나가 없으면 진행하지 않는다 — 사후 기억으로 재구성한 요약은 원본으로 인정하지 않는다.

- [ ] 세션 녹화본 (영상/음성) 또는 실시간 관찰 노트(타임코드 포함)
- [ ] 참가자 기본 정보 (연령대·기기·숙련도 — `PERSONA.md` 매핑에 필요)
- [ ] 테스트 대상 화면/시나리오 ID (`UT_SCENARIOS.md` 의 시나리오 코드와 매칭)

셋 중 하나라도 없으면: "원본 부재 — 인테이크 불가. {누락 항목} 확보 후 재요청" 으로 중단한다.

---

## Step 1: 원문 보존 (요약 금지)

`INTERVIEW_GUIDE.md` 의 규칙을 그대로 상속한다:

- 발화·행동을 해석·요약하지 않고 **원문 그대로 + 타임코드**로 기록한다.
- 녹화에 없는 내용을 추정해서 채우지 않는다 → 불확실하면 "확인 필요"로 표기한다.
- 관찰자의 해석(예: "사용자가 답답해 보였다")과 관찰된 사실(예: "동일 버튼을 3번 클릭 후 8초간 정지")을 분리해서 기록한다.

---

## Step 2: AI 관측 스키마로 정규화

`UT_OBSERVATION_SHEET.md` 와 **동일한 필드**로 변환한다. 필드명이 다르면 이후 `ut-calibrate.mjs` 가
자동으로 diff를 계산할 수 없으므로, 아래 매핑을 엄격히 따른다.

| AI 관측 필드 (UT_OBSERVATION_SHEET.md) | 실측 환산 방법 |
|---|---|
| 완료 여부 | 세션 종료 시점에 성공 조건 충족 여부 (녹화 기준 판정) |
| 총 소요 시간 | 시나리오 시작 타임코드 ~ 종료/이탈 타임코드 |
| 오류 횟수 | 잘못된 경로 진입·재시도·롤백 행동 횟수 (원문 근거와 함께) |
| 최장 머뭇거림 | 특정 요소 앞에서 정지 또는 반복 hover/스크롤한 최장 구간 |
| 이탈 지점 | 포기/이탈한 단계명 (없으면 "없음") |
| 스크린샷 | 녹화본 캡처 프레임 경로 (있으면) |

여기에 AI 쪽엔 없는 필드를 추가한다 (실측만 가능한 정보):

| 실측 전용 필드 | 설명 |
|---|---|
| 원문 발화 (think-aloud) | 타임코드와 함께 원문 인용 |
| 관찰자 메모 | 사실과 해석을 분리 표기 |

---

## Step 3: 페르소나 매핑 (추측 금지 — 애매하면 확인)

실제 참가자를 `refs/ux-research/PERSONA.md` 의 P1/P2/P3 중 어디에 매핑할지 판정한다.

- 판정 근거: 참가자의 서비스 숙련도·기기 사용 패턴·과업 접근 방식이 `PERSONA.md` 의 어떤 정의와 가장 가까운지
- 하나의 정의에 명확히 들어맞지 않으면 **새 페르소나를 여기서 만들지 않는다** — `PERSONA.md#미분류` 로 표기하고
  사용자에게 "이 참가자를 기존 페르소나에 매핑할지, PERSONA.md 갱신이 필요한지" 확인한다.
- 매핑 결과는 반드시 근거(참가자 발언/행동 인용)와 함께 기록한다.

---

## Step 4: 저장

`refs/ux-research/real-ut-sessions/{scenario-id}/{session-id}.md` 로 저장한다.
템플릿은 `refs/ux-research/REAL_UT_SESSION_LOG_TEMPLATE.md` 참조.

같은 `{scenario-id}` 아래 세션이 여러 개 쌓이면, `ut-calibrate.mjs` 가 자동으로
평균·분산을 계산해 AI 시뮬레이션과 비교할 수 있게 된다 (n=1일 때는 "일반화 불가" 경고 유지).

---

## Step 5: 신뢰도 등록 (collaboration-tracker)

`ux-research-sync.md` 의 신뢰도 규약을 그대로 따른다:

- 이 화면·시나리오에 대한 실측 데이터 → **🟢 검증** (1차, 표본 한계 명시: 세션 수 n, 참가자 구성)
- `refs/collaboration-tracker.md` 의 "실증 데이터 출처" 표에 세션 등록 (출처: real-ut-intake, 날짜, scenario-id)
- 세션 수가 늘어날수록 표본 한계 문구를 갱신한다 (예: "n=1 → 일반화 불가" → "n=5 → 잠정 일반화 가능, 외부 검증 필요")

---

## Step 6: 다음 단계 안내

```
✅ 실 UT 세션 인테이크 완료

📁 저장 위치: refs/ux-research/real-ut-sessions/{scenario-id}/{session-id}.md
🧑 페르소나 매핑: {P1/P2/P3 또는 미분류}
🟢 신뢰도: 검증 (n={세션 수})

⚡ 다음 단계:
  - 같은 scenario-id 로 AI 시뮬레이션이 이미 있다면 → `ut-calibrate.mjs` 실행해 gap 확인
  - 아직 없다면 → `ai-usability-test` 먼저 실행 후 캘리브레이션
  - 세션이 3개 미만이면 일반화 근거로 쓰지 말 것 (Executive Summary에 표본 한계 명시)
```

## 타 스킬과의 관계

| 스킬 | 관계 |
|---|---|
| `ai-usability-test` | 이 스킬이 저장한 실측 데이터와 **동일 scenario-id** 로 비교됨 (Step 2 스키마 공유) |
| `ux-research-sync` | 이 스킬의 산출물이 `ux-research-sync` 의 "1차 데이터" 소스 후보가 됨 (신뢰도 규약 상속) |
| `ut-calibrate.mjs` | 이 스킬의 산출물을 입력으로 받아 AI-실측 gap을 계산하고 파라미터 조정을 제안 |

## 한계

- 세션을 만들어내지 않는다 — 모집·동의·진행은 여전히 사람의 몫
- n<3인 상태에서의 캘리브레이션 제안은 참고용일 뿐, 확정적 파라미터 변경 근거로 쓰지 않는다
- 녹화/노트가 부실하면(타임코드 누락 등) Step 2 정규화 품질이 떨어지므로, 세션 진행 시점에
  최소한의 관측 스키마를 미리 공유해두는 것을 권장한다 (연구자에게 `UT_OBSERVATION_SHEET.md` 필드 사전 공유)
